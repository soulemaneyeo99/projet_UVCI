1. Introduction
Ce  document  constitue  le  Cahier  des  Charges  (CDC)  du  projet  de  fin  d'annee  intitule 
"Application de Gestion des Heures des Enseignants" developpe dans le cadre de l'Universite 
Virtuelle de Cote d'Ivoire (UVCI). Il definit de maniere exhaustive les besoins fonctionnels, les 
exigences techniques, l'architecture logicielle et le planning de realisation du projet.
L'objectif est  de fournir une reference  commune  a toute  l'equipe de developpement (5  
membres) pour garantir la coherence, la qualite et la livraison dans les delais impartis de deux 
semaines.
2. Contexte et Problematique
2.1 Contexte
L'Universite Virtuelle de Cote d'Ivoire (UVCI) est un etablissement d'enseignement superieur  
dont  le  modele  pedagogique  repose  sur  la  production  et  la  diffusion  de  ressources  
numeriques. Les activites des enseignants vont au-dela du cours traditionnel : elles incluent la 
conception  de  ressources  pedagogiques  numeriques,  la  creation  de  sequences  
pedagogiques, l'elaboration de quiz et d'evaluations, la production d'activites interactives et la 
mise a jour reguliere des contenus.
2.2 Problematique
Actuellement, la gestion de ces activites repose sur des fichiers Excel et des procedures  
manuelles, ce qui engendre plusieurs dysfonctionnements majeurs :
• Erreurs frequentes dans le calcul des volumes horaires
• Absence de traçabilite des modifications apportees aux donnees
• Difficultes de consolidation et de centralisation des informations
• Retards dans la preparation des etats de paiement des enseignants
• Manque de visibilite globale sur la charge de travail des enseignants
Face a ces problemes, il est necessaire de developper une application web robuste, securisee 
et ergonomique pour automatiser et fiabiliser ces processus.
3. Objectifs du Projet
3.1 Objectif General
Developper une application web full-stack permettant la gestion automatisee et securisee des 
heures d'enseignement et des activites pedagogiques des enseignants de l'UVCI.
3.2 Objectifs Specifiques
• Permettre l'enregistrement et la gestion complete des enseignants

4.2 Modules Fonctionnels
L'application est organisee autour des modules suivants :
Module 1 - Gestion des Enseignants
• Creation, modification et suppression d'un enseignant
• Consultation de la liste des enseignants avec filtres
• Informations gerees : nom, prenom, grade (Assistant / Maitre-Assistant / Professeur), statut 
(Permanent / Vacataire), departement, taux horaire, email, telephone

• Gestion des cours : intitule, filiere, niveau (L1 a M2), semestre
• Nombre d'heures et credits associes au cours
• Association enseignant-cours
Module 3 - Gestion des Ressources Pedagogiques
• Enregistrement des ressources : contenus textuels, videos, documents, quiz, activites 
interactives, evaluations
• Structuration des cours en sequences pedagogiques (Niveau 1, 2 et 3)

entre le frontend, le backend et la base de donnees. Un service de reconnaissance faciale est 
integre cote serveur pour securiser l'authentification.

5.2 Technologies et Outils
Frontend - Next.js
• Framework React avec rendu hybride (SSR + CSR)
• Gestion des routes avec le App Router de Next.js 14+
• UI : Tailwind CSS pour le style, Shadcn/ui ou Ant Design pour les composants
• Gestion d'etat : Zustand ou React Context API
• Appels API : Axios ou Fetch natif
• Generation de graphiques : Recharts ou Chart.js
Backend - Python FastAPI
• Framework web Python moderne, rapide et asynchrone
• Documentation API automatique avec Swagger UI
• Validation des donnees avec Pydantic
• Gestion de l'authentification JWT (JSON Web Tokens)
• Librairies : SQLAlchemy (ORM), Alembic (migrations), Passlib (hachage mots de passe)
• Export PDF : ReportLab ou WeasyPrint ; Export Excel : openpyxl
FaceAPI - Reconnaissance Faciale
• Integration dans le backend Python pour la verification biometrique
• Bibliotheque recommandee : face_recognition (dlib) ou DeepFace
• Flux : capture de la photo en frontend → envoi base64 → verification cote serveur
• Utilisation : etape de second facteur d'authentification apres le mot de passe
• Stockage des encodages faciaux en base de donnees (champ binaire ou JSON)
Base de Donnees
• SQLite en phase de developpement (aucune installation requise)
• Migration prevue vers PostgreSQL en production
• Gestion des migrations avec Alembic

7. Exigences Non Fonctionnelles
7.1 Performance
• Temps de reponse API inferieur a 500ms pour les requetes courantes
• Interface utilisateur reactive et fluide (SPA avec Next.js)
• Traitement de la reconnaissance faciale en moins de 3 secondes
7.2 Disponibilite et Fiabilite
• Application accessible via tout navigateur web moderne (Chrome, Firefox, Edge)
• Interface responsive compatible desktop et tablette
• Sauvegarde automatique des donnees
7.3 Securite
• Authentification obligatoire pour acceder a toutes les fonctionnalites
• Donnees transmises via HTTPS en production
• Mots de passe haches avec bcrypt (facteur de cout >= 12)
• Encodages faciaux stockes de maniere securisee
• Journalisation des actions sensibles (audit log)

10. Conclusion
Ce cahier des charges definit le cadre complet du projet "Application de Gestion des Heures  
des Enseignants" de l'UVCI. La combinaison des technologies Next.js, Python FastAPI,  
FaceAPI  et SQLite  permet  de  construire  une  solution  moderne,  securisee  et  evolutive,  
repondant aux besoins identifies.
L'equipe de 5 membres disposant de deux semaines devra respecter le planning etabli et  
assurer une communication continue pour garantir la coherence entre le frontend et le  
backend. Une attention particuliere devra etre portee a l'integration du module FaceAPI et au 
calcul automatique des volumes horaires, qui constituent les deux points techniques les plus  
specifiques du projet.
Ce document servira de reference tout au long du developpement et sera mis a jour si  
necessaire pour refleter les evolutions du projet.
